# Requirements Document

## Introduction

MowisAI currently runs only on Linux because `agentd` depends on Linux-specific kernel features (overlayfs, cgroups v2, Linux namespaces, Unix domain sockets) and `mowis-gui/src/backend.rs` hard-codes Unix socket paths and Unix-only Rust APIs. This feature makes MowisAI work seamlessly on macOS and Windows by adopting a Docker-Desktop-style architecture: the native GUI runs on every platform, while `agentd` runs inside a lightweight Linux VM that is launched silently and automatically on first run. No user-visible prerequisites, no Homebrew, no WSL2 manual setup — just download and double-click.

**Platform strategy:**
- **Linux** — `agentd` runs directly, native speed, no VM.
- **macOS 10.15+** — `agentd` runs inside a microVM via Apple's built-in `Virtualization.framework`. Fallback: bundled static QEMU.
- **Windows 10 2004+** — `agentd` runs inside WSL2 (built into Windows). Fallback: bundled static QEMU.

---

## Glossary

- **VM_Launcher** — The platform-specific Rust component responsible for starting and stopping the Linux VM that hosts `agentd` on macOS and Windows.
- **macOS_Launcher** — The VM_Launcher implementation for macOS, using `Virtualization.framework` via FFI.
- **WSL2_Launcher** — The VM_Launcher implementation for Windows, using `wsl.exe` to import and run a custom Alpine WSL2 distribution.
- **QEMU_Launcher** — The fallback VM_Launcher for macOS < 10.15 and Windows without WSL2, using a bundled static QEMU binary.
- **Socket_Bridge** — The component that connects `mowis-gui` to `agentd` using the most secure channel available on the current platform (Unix socket, vsock, named pipe, or authenticated TCP).
- **Auth_Token** — A 256-bit cryptographically random secret generated by `agentd` on each startup, written to a user-readable-only file (`chmod 600`), and required on every TCP connection. Used only on Windows and QEMU fallback paths.
- **Backend** — The `mowis-gui` component (`mowis-gui/src/backend.rs`) that manages daemon lifecycle and socket communication.
- **Alpine_Image** — The compressed Alpine Linux disk image (~50 MB) bundled with the application, containing a pre-built static `agentd` binary.
- **App_Bundle** — The platform-specific distributable package (`.dmg` on macOS, `.exe` installer on Windows, native binary on Linux).
- **First_Run** — The initial launch of MowisAI on a machine where the VM has not yet been set up.
- **Socket_Path** — The filesystem path or address used by `mowis-gui` to connect to `agentd`. Resolved at runtime per platform: Unix socket on Linux/macOS, Windows named pipe or authenticated TCP on Windows.
- **agentd** — The Linux-only AI agent orchestration daemon that uses overlayfs, cgroups, namespaces, and Unix sockets.
- **mowis-gui** — The native egui desktop GUI, compiled natively for each target platform.
- **Workspace** — The Cargo workspace containing `agentd`, `agentd-protocol`, `runtime`, and `mowis-gui`.

## Requirements

### Requirement 1: Platform-Aware Daemon Lifecycle

**User Story:** As a user on macOS or Windows, I want MowisAI to start the Linux daemon automatically without any manual setup, so that I can use the application immediately after installation.

#### Acceptance Criteria

1. WHEN the application starts on Linux, THE Backend SHALL launch `agentd` directly as a child process using the existing Unix socket mechanism.
2. WHEN the application starts on macOS 10.15 or later, THE macOS_Launcher SHALL start a Linux microVM using `Virtualization.framework` and boot `agentd` inside it.
3. WHEN the application starts on Windows 10 version 2004 or later with WSL2 available, THE WSL2_Launcher SHALL import the Alpine_Image as a WSL2 distribution and start `agentd` inside it.
4. IF `Virtualization.framework` is unavailable on macOS, THEN THE QEMU_Launcher SHALL start `agentd` using the bundled static QEMU binary.
5. IF WSL2 is unavailable on Windows, THEN THE QEMU_Launcher SHALL start `agentd` using the bundled static QEMU binary.
6. WHEN the VM_Launcher starts a VM for the first time, THE VM_Launcher SHALL display a progress message "Setting up AI engine (first time only)..." in the GUI for the duration of the setup.
7. WHEN the VM_Launcher starts a VM on subsequent launches, THE VM_Launcher SHALL reuse the cached VM state so that startup completes within 5 seconds.
8. WHEN the application exits, THE VM_Launcher SHALL stop the VM and release all associated resources.

---

### Requirement 2: Secure Socket Bridge

**User Story:** As a developer and as a security-conscious enterprise customer, I want `mowis-gui` to communicate with `agentd` through the most secure channel available on each platform, so that no unauthorized process can send commands to the daemon.

#### Acceptance Criteria

1. WHEN running on Linux, THE Socket_Bridge SHALL use a Unix domain socket at a path under the user's runtime directory (e.g. `$XDG_RUNTIME_DIR/agentd.sock`) with file permissions set to `0600` so that only the owning user's processes can connect.
2. WHEN running on macOS with `Virtualization.framework`, THE Socket_Bridge SHALL use virtio-vsock to tunnel the `agentd` Unix socket from inside the VM to the host, exposing it as a Unix domain socket with `0600` permissions. No TCP port is opened.
3. WHEN running on Windows with WSL2, THE Socket_Bridge SHALL expose the `agentd` Unix socket as a Windows named pipe (`\\.\pipe\MowisAI\agentd`) secured with a Windows ACL that permits only the current user's SID, so that no other user or process can connect.
4. WHEN running on Windows without WSL2, OR when using the QEMU fallback on any platform, THE Socket_Bridge SHALL use TCP loopback (`127.0.0.1` with a randomly chosen ephemeral port) protected by an Auth_Token, following the same security model used by Jupyter Notebook and VS Code Remote.
5. WHEN the Auth_Token mechanism is active, `agentd` SHALL generate a 256-bit cryptographically random token at startup, write it to `~/.mowisai/auth-token` with permissions `0600`, and reject any connection whose first message does not contain the correct token.
6. WHEN the Auth_Token mechanism is active, THE Backend SHALL read the token from `~/.mowisai/auth-token` and include it in the first message of every new connection before sending any other data.
7. THE Backend SHALL select the Socket_Path and authentication method at runtime based on the current platform without requiring user configuration.
8. WHEN the socket connection is lost due to a VM or daemon crash, THE Backend SHALL attempt to reconnect up to 5 times with a 1-second delay between attempts before reporting a connection failure to the GUI.
9. THE Socket_Bridge SHALL preserve the full JSON-RPC protocol semantics so that all 30+ request types function identically regardless of the underlying transport.

### Requirement 3: mowis-gui Compile Compatibility

**User Story:** As a developer, I want `mowis-gui` to compile without errors on macOS and Windows, so that the GUI binary can be built natively for each target platform.

#### Acceptance Criteria

1. THE mowis-gui crate SHALL compile without errors on `x86_64-apple-darwin`, `aarch64-apple-darwin`, `x86_64-pc-windows-msvc`, and `x86_64-unknown-linux-gnu` targets.
2. WHEN compiling on Windows, THE mowis-gui crate SHALL use TCP socket communication (`tokio::net::TcpStream`) with Auth_Token authentication instead of `tokio::net::UnixStream`.
3. WHEN compiling on Linux or macOS, THE mowis-gui crate SHALL use Unix domain socket communication (`tokio::net::UnixStream`).
4. THE mowis-gui crate SHALL guard all uses of `std::os::unix` APIs with `#[cfg(unix)]` attributes so that Windows builds do not reference Unix-only symbols.
5. THE mowis-gui crate SHALL NOT introduce any new platform-specific dependencies that require system libraries not available in a standard Rust toolchain installation.

---

### Requirement 4: macOS VM Launcher

**User Story:** As a macOS user, I want the application to use Apple's built-in virtualisation engine, so that no third-party software needs to be installed.

#### Acceptance Criteria

1. THE macOS_Launcher SHALL invoke `Virtualization.framework` APIs via Rust FFI or a thin Swift/Objective-C shim compiled into the App_Bundle.
2. WHEN booting the Alpine_Image, THE macOS_Launcher SHALL configure the VM with at least 512 MB of RAM and 1 virtual CPU.
3. THE macOS_Launcher SHALL expose the `agentd` socket to the host using virtio-vsock, presenting it as a Unix domain socket with `0600` permissions on the host side. No TCP port SHALL be opened on macOS.
4. WHEN the Alpine_Image boots for the first time, THE macOS_Launcher SHALL write a VM state snapshot to the user's application support directory so that subsequent boots skip the full boot sequence.
5. IF the VM state snapshot is corrupt or missing, THEN THE macOS_Launcher SHALL perform a full boot from the Alpine_Image and create a new snapshot.
6. THE macOS_Launcher SHALL run without requiring administrator privileges or any entitlement beyond `com.apple.security.virtualization`.

---

### Requirement 5: Windows WSL2 Launcher

**User Story:** As a Windows user, I want the application to use WSL2, which is already built into Windows 10 2004+, so that no additional software needs to be installed.

#### Acceptance Criteria

1. WHEN WSL2 is available, THE WSL2_Launcher SHALL import the Alpine_Image as a WSL2 distribution named `MowisAI` using `wsl.exe --import`.
2. THE WSL2_Launcher SHALL run `wsl.exe` commands as the current user without triggering a UAC elevation prompt.
3. WHEN the `MowisAI` WSL2 distribution is already imported, THE WSL2_Launcher SHALL skip the import step and start `agentd` directly.
4. THE WSL2_Launcher SHALL expose the `agentd` Unix domain socket from inside WSL2 as a Windows named pipe (`\\.\pipe\MowisAI\agentd`) secured with a Windows ACL permitting only the current user's SID. No TCP port SHALL be opened when WSL2 is available.
5. WHEN the WSL2_Launcher starts `agentd`, THE WSL2_Launcher SHALL pass the Unix socket path as a command-line argument and configure the named pipe bridge to forward traffic to that socket.
6. IF the `MowisAI` WSL2 distribution becomes corrupted, THEN THE WSL2_Launcher SHALL unregister the distribution and re-import it from the bundled Alpine_Image.

---

### Requirement 6: QEMU Fallback Launcher

**User Story:** As a user on an older macOS version or a Windows machine without WSL2, I want MowisAI to still work using a bundled QEMU binary, so that I am not blocked by OS version requirements.

#### Acceptance Criteria

1. THE App_Bundle SHALL include a statically linked QEMU binary of no more than 25 MB for the host architecture.
2. WHEN the QEMU_Launcher starts, THE QEMU_Launcher SHALL boot the Alpine_Image using the bundled QEMU binary with KVM acceleration on Linux, HVF acceleration on macOS, and WHPX or TCG acceleration on Windows.
3. THE QEMU_Launcher SHALL forward the `agentd` Unix domain socket from inside the QEMU VM to a TCP loopback port on the host using QEMU's built-in port forwarding (`hostfwd`), protected by the Auth_Token mechanism defined in Requirement 2.
4. WHEN the QEMU VM is running, THE QEMU_Launcher SHALL monitor the QEMU process and restart it if it exits unexpectedly.
5. THE QEMU_Launcher SHALL pass the `agentd` socket path and forwarded TCP port as environment variables to the Alpine_Image init script.

---

### Requirement 7: Alpine Linux Image

**User Story:** As a developer, I want a minimal, pre-built Alpine Linux image that contains everything `agentd` needs, so that VM boot time is fast and the bundle size is small.

#### Acceptance Criteria

1. THE Alpine_Image SHALL be a compressed disk image of no more than 100 MB when stored in the App_Bundle.
2. THE Alpine_Image SHALL contain a statically linked `agentd` binary compiled with `--target x86_64-unknown-linux-musl` for x86_64 hosts and `--target aarch64-unknown-linux-musl` for ARM64 hosts (Apple Silicon).
3. THE Alpine_Image SHALL contain an init script that starts `agentd socket --path /tmp/agentd.sock` on boot and restarts it if it exits.
4. THE Alpine_Image SHALL contain the `skopeo` binary required by `agentd` for container image pulls.
5. WHEN the Alpine_Image init script starts, THE Alpine_Image SHALL configure the network interface so that `agentd` can make outbound HTTPS requests.

---

### Requirement 8: Static agentd Binary Build

**User Story:** As a developer, I want a reproducible build process for the static `agentd` binary, so that the binary embedded in the Alpine_Image is always up to date with the source.

#### Acceptance Criteria

1. THE Workspace SHALL provide a CI build target that compiles `agentd` with `cargo build --release --target x86_64-unknown-linux-musl` and produces a fully statically linked binary.
2. THE Workspace SHALL provide a CI build target that compiles `agentd` with `cargo build --release --target aarch64-unknown-linux-musl` for Apple Silicon hosts.
3. WHEN building the static `agentd` binary, THE build process SHALL set `RUSTFLAGS="-C target-feature=+crt-static"` to ensure no dynamic libc dependency.
4. THE agentd crate's `Cargo.toml` SHALL gate all Linux-specific dependencies (`nix`, `signal-hook`) under `[target.'cfg(target_os = "linux")'.dependencies]` so that the crate graph resolves correctly on all platforms.
5. THE agentd crate SHALL compile on non-Linux hosts without errors by guarding all uses of Linux-specific APIs (`nix::mount`, `nix::sched`, `signal_hook`) with `#[cfg(target_os = "linux")]` attributes.

---

### Requirement 9: App Bundle Packaging

**User Story:** As a user, I want to download a single file and double-click it to install MowisAI, so that there are no manual steps.

#### Acceptance Criteria

1. THE macOS App_Bundle SHALL be a signed `.dmg` file containing `MowisAI.app` with the `mowisai` GUI binary, `agentd.linux` static binary, `alpine.img` disk image, and `vm_launcher` helper binary.
2. THE Windows App_Bundle SHALL be a single `.exe` installer that extracts the `mowisai.exe` GUI binary, `agentd.linux` static binary, `alpine.img` disk image, and `wsl_launcher.exe` helper binary to the user's application directory without requiring administrator privileges.
3. THE Linux App_Bundle SHALL be a single statically linked binary that includes `agentd` as an embedded resource and extracts it to a temporary directory on first run.
4. WHEN the App_Bundle is launched for the first time, THE App_Bundle SHALL verify the integrity of the bundled `agentd.linux` binary and `alpine.img` using SHA-256 checksums stored in the bundle.
5. IF the integrity check fails, THEN THE App_Bundle SHALL display an error message identifying the corrupt file and exit without starting the VM.

---

### Requirement 10: First-Run User Experience

**User Story:** As a first-time user, I want to see clear progress feedback during the initial setup, so that I know the application is working and not frozen.

#### Acceptance Criteria

1. WHEN the application detects a First_Run condition, THE Backend SHALL emit a `DaemonStarting` event to the GUI before launching the VM_Launcher.
2. WHEN the VM_Launcher is running, THE mowis-gui SHALL display the message "Setting up AI engine (first time only)..." with an animated progress indicator in the landing view.
3. WHEN the VM_Launcher completes setup and `agentd` is reachable, THE Backend SHALL emit a `DaemonStarted` event and THE mowis-gui SHALL dismiss the progress indicator and show the normal landing view.
4. IF the VM_Launcher fails to start `agentd` within 30 seconds, THEN THE Backend SHALL emit a `DaemonFailed` event with a human-readable error message and THE mowis-gui SHALL display the error with a "Retry" button.
5. THE mowis-gui SHALL NOT block the egui render loop during VM startup; all VM_Launcher operations SHALL run on the existing background tokio runtime thread.

---

### Requirement 11: CI/CD Pipeline

**User Story:** As a developer, I want automated builds and tests for all three platforms, so that regressions are caught before release.

#### Acceptance Criteria

1. THE Workspace SHALL include a GitHub Actions workflow that builds `mowis-gui` natively on `ubuntu-latest`, `macos-latest`, and `windows-latest` in a single matrix job.
2. THE GitHub Actions workflow SHALL build the static `agentd` binary for `x86_64-unknown-linux-musl` and `aarch64-unknown-linux-musl` targets as part of the macOS and Windows jobs.
3. THE GitHub Actions workflow SHALL run `cargo test --workspace` on `ubuntu-latest` and verify that all existing tests pass.
4. THE GitHub Actions workflow SHALL run `cargo clippy --workspace -- -D warnings` on all three platforms and fail the build if any warnings are emitted.
5. WHEN a Git tag matching `v*` is pushed, THE GitHub Actions workflow SHALL produce release artifacts: `MowisAI-macOS.dmg`, `MowisAI-Windows.exe`, and `MowisAI-Linux.tar.gz`.

### Requirement 13: Connection Security Model

**User Story:** As an enterprise security auditor, I want to verify that no unauthorized process can communicate with the `agentd` daemon, so that I can certify MowisAI for deployment in regulated environments.

#### Acceptance Criteria

1. THE system SHALL use the strongest available IPC mechanism on each platform, in priority order: Unix domain socket (Linux/macOS) → virtio-vsock (macOS VM) → Windows named pipe with ACL (Windows WSL2) → TCP loopback with Auth_Token (QEMU fallback).
2. ON Linux, THE Unix domain socket SHALL have file permissions `0600` so that only the process owner can connect; no network port SHALL be opened.
3. ON macOS, THE virtio-vsock channel SHALL be accessible only to the host process that owns the VM; no TCP port SHALL be opened.
4. ON Windows with WSL2, THE Windows named pipe SHALL be secured with a DACL granting `GENERIC_READ | GENERIC_WRITE` only to the current user's SID; no TCP port SHALL be opened.
5. WHEN TCP loopback is used as a fallback, THE Auth_Token SHALL be a minimum of 256 bits of output from a cryptographically secure random number generator (Rust `rand::rngs::OsRng`).
6. WHEN TCP loopback is used as a fallback, `agentd` SHALL close the connection immediately and log a warning if the first message does not contain the correct Auth_Token, without processing any part of the message payload.
7. THE Auth_Token file SHALL be written with permissions `0600` on Unix and with a Windows ACL granting access only to the current user on Windows, so that other users on a shared machine cannot read it.
8. THE TCP loopback port SHALL be chosen randomly from the ephemeral range (49152–65535) at each `agentd` startup and written alongside the Auth_Token so that port scanners cannot predict it.



**User Story:** As a developer, I want the JSON-RPC protocol to serialize and deserialize correctly across the socket bridge on all platforms, so that no messages are lost or corrupted when forwarded through TCP on Windows.

#### Acceptance Criteria

1. THE agentd-protocol crate SHALL define all request and response types with `#[derive(Serialize, Deserialize)]` and no platform-specific fields.
2. FOR ALL valid `SocketRequest` values, serializing to JSON and then deserializing SHALL produce an equivalent `SocketRequest` (round-trip property).
3. FOR ALL valid `SocketResponse` values, serializing to JSON and then deserializing SHALL produce an equivalent `SocketResponse` (round-trip property).
4. WHEN a `SocketRequest` is forwarded through the Socket_Bridge TCP tunnel on Windows, THE Socket_Bridge SHALL preserve the exact byte sequence of the JSON payload including newline delimiters.
5. THE agentd-protocol crate SHALL compile without warnings on `x86_64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`, and `x86_64-pc-windows-msvc` targets.
