## 0. What changed from v1

v1 was correct in shape but had three real gaps the user (correctly)
flagged:

1. **Conductor went silent during execution.** v1's Conductor finished
   when the plan was approved and only woke up on new user messages.
   v2 keeps the Conductor **alive for the whole conversation** and lets
   the user inject "also do X" mid-run.
2. **Captain went silent after `PlanCompleted`.** v1 spec used
   `async fn run(self) -> CaptainOutcome` — return-then-die. v2 makes
   Captain a long-running actor for the whole conversation; the user
   can query, the Conductor can hot-patch new tasks into it.
3. **No real-time stream from agents to anyone.** v1 emitted metadata
   events (`CrewProgress { tool, round }`) but never specified how the
   work the agent actually did surfaces to the Captain LLM or the UI.
   v2 introduces `CrewToolSummary { agent_id, text, tool_name, success }`
   — one human-readable sentence per tool call, deterministically
   generated by a per-tool template registry (no LLM cost).

Five smaller refinements:

4. **Hot-patch vs scope-change** decision is an explicit LLM
   classification step inside the Conductor (§5.5), not a hand-wave.
5. **Critic** is advisory; the user is the only approval gate. v1
   was confused on this (§6).
6. **Conductor digest of summaries:** Conductor subscribes to the
   summary stream but reads a *digest* on next invocation, not the
   raw stream. This avoids reinventing the context-bloat problem at
   the Conductor tier (§5.4).
7. **New `ConversationEnded` event** triggers Captain shutdown +
   sandbox cleanup (§9, §7.8). v1 had no defined endpoint for the
   sandbox lifecycle.
8. **UI streams (new §13)** define which streams the user sees by
   default and which are click-to-reveal.

Plan format (§4), providers (§10), agentd shrinkage (§14) are
unchanged from v1.

## 1. Architecture: the four tiers

The system is a **single host process** (`mowisd`) talking to a **single
guest VM** (`mowis-executor`) over AF_VSOCK. All four LLM tiers run as
async tasks **inside the host process**. Tool calls cross vsock to the
guest. Nothing else crosses vsock.

```
                                  HOST (mowisd) — one process per conversation
   ┌──────────────────────────────────────────────────────────────────────┐
   │                                                                      │
   │   User <───── TUI / HTTP API ──── Conductor (flagship LLM, actor)    │
   │      ▲                                ▲                              │
   │      │                                │ mid-run inject (§5.5)        │
   │      │ chat                           │                              │
   │      │                                │ summary digest on wake (§5.4)│
   │      │                                │                              │
   │      │            ┌───────────────────┘                              │
   │      │            │   drafts Plan (multi-file)                       │
   │      │            ▼                                                  │
   │      │      .mowis/plans/<id>/                                       │
   │      │            │                                                  │
   │      │   ┌────────┴────────┐                                         │
   │      │   │ in parallel     │                                         │
   │      │   ▼                 ▼                                         │
   │      │  User reviews  Critic (flagship, blind, one-shot, no tools)   │
   │      │   │                 │                                         │
   │      │   │                 └── posts verdict to chat ──► Conductor   │
   │      │   │                     (Conductor may revise; loop ≤4)       │
   │      │   ▼                                                           │
   │      │  User clicks Approve  ─────────────────────────────► Captain  │
   │      │                                                       (actor) │
   │      │                                                          │    │
   │      │              spawn N Crews ────┬─────────────┐               │
   │      │              ▼                  ▼             ▼               │
   │      │         Crew #1 (fast)    Crew #2 (fast) ... Crew #N (fast)   │
   │      │              │                  │             │               │
   │      │              └──── CrewToolSummary events ────┘               │
   │      │                            │                                  │
   │      │                            ▼                                  │
   │      │                  Event bus (broadcast)                        │
   │      │                  ├─► Captain LLM (consumes, may steer)        │
   │      │                  ├─► Conductor digest buffer (§5.4)           │
   │      │                  └─► UI streams (§13)                         │
   │      └──────────────────────────────────────────────────────────────-│
   │                                                                      │
   └──────────────────────────────────────────────────────────────────────┘
                                      │
                                 vsock (cid:port)
                                      │
                                      ▼
   ┌──────────────────────────────────────────────────────────────────────┐
   │                       GUEST VM (mowis-executor)                      │
   │  - one Level-1 sandbox per conversation                              │
   │  - one Level-2 agent-overlay per Crew                                │
   │  - tool registry (filesystem, git, docker, k8s, shell, plugins,      │
   │    skills, ...)                                                      │
   └──────────────────────────────────────────────────────────────────────┘
```

### 1.1 Tier responsibilities

| Tier | Spawned | Lifetime | Model tier | Tools |
|---|---|---|---|---|
| **Conductor** | Once per conversation | **Whole conversation** | Flagship | Skills, plugins, repo-read (`grep`, `read_file`, `list_files`). Plus internal: `inject_task_to_captain`, `pause_captain`, `replan`. **No** shell, **no** write, **no** git. |
| **Critic** | Once per plan version | One-shot per version | Flagship | **None.** Pure text in → structured verdict out. |
| **Captain** | After user approves the first plan version | **Whole conversation** (idle when no plan running) | Mid (Sonnet-class) | Orchestration only: `create_sandbox`, `start_crew`, `get_crew_status`, `pause_crew`, `cancel_crew`, `merge_overlay`, `discard_overlay`, `replan_subgraph`. **Does not touch user files.** |
| **Crew** | One per Plan task, spawned by Captain | Until task completes | Fast (Haiku / 4o-mini / Flash / Llama-8B) | Everything: filesystem, git, docker, k8s, http, shell, all plugins, all skills. |

### 1.2 Sandbox topology

(Decided: **One sandbox per conversation, N agent overlays.**)

- **Conversation start** → Captain (after first `PlanApproved`) asks
  guest to `CreateSandbox` exactly once (Level-1 CoW over the project
  rootfs). `sandbox_id` is held by Captain for the life of the
  conversation.
- **Per Crew spawn** → Captain calls `CreateAgentOverlay
  { parent_sandbox_id, agent_id }` (§11) which creates a Level-2
  overlay over the conversation sandbox. The Crew's tool calls
  reference `agent_id`; the executor routes them to that overlay.
- **On Crew success** → Captain calls `MergeAgentOverlay { agent_id }`
  to fold the agent layer into the conversation sandbox. Merges happen
  in task-dependency order; **siblings merge with a tree-merge**
  (log₂ N rounds, see `agentd/src/orchestration/merge_worker.rs` for
  the algorithm to port).
- **On Crew failure** → Captain calls `DiscardAgentOverlay { agent_id }`
  and decides (replan / retry / abort) per §7.6.
- **On `ConversationEnded`** → Captain calls `DestroySandbox` and
  shuts itself down (§7.8).

### 1.3 What does *not* change

- The `mowis-protocol` wire format is stable (only **additions** allowed,
  see §11).
- The guest (`mowis-executor`) sandbox primitives are stable. The new
  endpoints in §11 are thin wrappers around the existing sandbox module.
- The vsock transport, initrd builder, qemu launcher are unchanged.

## 3. Why this shape

- **Orchestration is host-side, not guest-side.** LLM API calls have
  egress, secrets, and need observable tracing. The guest VM is the
  *target environment*, not an LLM client. Keep all
  `reqwest`-to-OpenAI / Anthropic / Vertex traffic on the host.
- **Conductor/Critic/Captain/Crew share one tokio runtime.** No
  multi-process IPC, no JSON-RPC stdin. They communicate over a
  `tokio::sync::broadcast` event bus plus per-actor mpsc command
  channels. Crash isolation is acceptable at the conversation level —
  if one Crew panics, we surface it on the bus and let Captain decide.
- **Tools never run in the host process.** Every tool call goes over
  vsock to `mowis-executor`. This is enforced by the `ToolGateway`
  (§12.2); there is no fall-through local execution.
- **Long-running actors, not run-to-completion futures.** Conductor
  and Captain are long-lived `Actor`s with explicit `Command` enums.
  This is the central shape change from v1. Without it, "mid-run
  injection" is impossible to implement cleanly.
- **One summary, one number, one stream.** Every Crew tool call emits
  exactly one `CrewToolSummary`. Captain LLM, Conductor digest, and UI
  all subscribe to the same broadcast — there is no parallel "raw vs
  summarised" universe to keep in sync.

## 4. `plan.rs` — the multi-file Plan format

### 4.1 On-disk layout

A Plan is a directory, not a file. Stored at `.mowis/plans/<plan_id>/`
where `<plan_id>` is `YYYYMMDDTHHMMSSZ-<6hex>` (UTC second + 6 random
hex chars).

```
.mowis/plans/<plan_id>/
├── plan.toml                # high-level metadata (see §4.2)
├── overview.md              # Conductor's prose: goal, approach, constraints
├── tasks.toml               # task graph (nodes, deps, model-tier hints) §4.3
├── sandbox.toml             # sandbox topology spec §4.4
├── models.toml              # model assignment per tier+task §4.5
├── tools.toml               # tool whitelist deltas vs tier default §4.6
├── critic/
│   ├── v1.md                # critic verdict on plan v1 (markdown)
│   ├── v1.toml              # structured verdict (approve | revise | block) §6.4
│   ├── v2.md
│   └── v2.toml              # only present if Conductor revised
├── history/
│   ├── v1/                  # snapshot of overview.md, tasks.toml, etc. at v1
│   ├── v2/                  # …at v2 after critic feedback
│   └── …
└── status.toml              # runtime state: which version is current,
                             # captain progress, crew statuses (§7.5)
```

The Conductor writes everything except `critic/` (Critic owns that) and
`status.toml` (Captain owns that, append-only). All files are
human-readable. TOML for structured, Markdown for prose.

### 4.2 `plan.toml` schema

```toml
plan_id      = "20260520T143155Z-a4f8c1"
created_at   = "2026-05-20T14:31:55Z"
conversation = "<conversation_id>"
current_version = 2          # which history/vN/ is live
status       = "draft"       # draft | awaiting_user | approved | running | done | aborted
user_goal    = """
multi-line verbatim user message that triggered this plan
"""
```

### 4.3 `tasks.toml` schema

```toml
[[task]]
id          = "t1"                          # short stable id
title       = "Add /healthz endpoint"
description = """
free-form, what the crew should accomplish
"""
deps        = []                            # task ids this depends on
model_tier  = "fast"                        # fast | mid | flagship (default fast)
tool_budget = 40                            # max tool calls before Captain steps in
files_hint  = ["src/server/routes.rs"]      # advisory; crew may touch others

[[task]]
id          = "t2"
title       = "Add unit test for /healthz"
deps        = ["t1"]
model_tier  = "fast"
tool_budget = 20
```

Cycles are an error. The Conductor must produce a DAG. `plan.rs`
validates this at load time.

### 4.4 `sandbox.toml` schema

```toml
image_rootfs = ".mowis-cache/rootfs/ubuntu-24.04"
ram_mb       = 8192
cpu_millis   = 4000

[overlays]
ram_mb       = 1024
cpu_millis   = 1000
```

### 4.5 `models.toml` schema

```toml
[tier.conductor]
provider = "anthropic"
model    = "claude-opus-4-7"

[tier.critic]
provider = "anthropic"
model    = "claude-opus-4-7"

[tier.captain]
provider = "anthropic"
model    = "claude-sonnet-4-6"

[tier.crew]
provider = "anthropic"
model    = "claude-haiku-4-5-20251001"

# Optional per-task override
[task.t1]
tier     = "mid"
```

### 4.6 `tools.toml` schema

```toml
[crew.allow_extra]
plugins   = ["my-internal-deploy-script"]

[crew.deny]
shell     = ["rm -rf"]  # exact-prefix denylist on arg vector
```

Deltas only — never restate the whole whitelist. The runtime composes
`tier_default ∪ allow_extra ∖ deny`.

### 4.7 `plan.rs` public API

```rust
pub struct Plan { /* parsed in-memory representation */ }

impl Plan {
    pub fn new_draft(plan_id: PlanId, user_goal: &str, conversation_id: &str) -> Self;
    pub fn load(plans_dir: &Path, plan_id: &PlanId) -> Result<Self>;
    pub fn save(&self) -> Result<()>;                  // writes the current version files
    pub fn snapshot_to_history(&mut self) -> Result<()>;
    pub fn revise(&mut self, conductor_output: ConductorRevision) -> Result<()>;

    pub fn validate(&self) -> Result<(), PlanError>;   // DAG, model refs, schema
    pub fn task_graph(&self) -> &TaskGraph;
    pub fn sandbox_spec(&self) -> &SandboxConfig;
    pub fn model_for(&self, tier: Tier, task: Option<&TaskId>) -> ModelRef;

}

pub struct PlanId(pub String);
pub enum Tier { Conductor, Critic, Captain, Crew }

pub struct TaskGraph { pub tasks: Vec<TaskNode> }

    pub id: TaskId, pub title: String, pub description: String,

}
```

`Plan::save` is atomic per-file: write to `…tmp` then rename. Never
leave partial files.

## 5. `conductor.rs` — user-facing planner (long-running actor)

### 5.1 Shape

The Conductor is an **actor** that owns its LLM client, conversation
history, current plan id (if any), and a digest buffer of recent Crew
summaries. It is alive for the whole conversation.

```rust
pub struct Conductor {
    cfg:      Arc<OrchConfig>,
    bus:      EventBus,
    cmd_rx:   mpsc::Receiver<ConductorCommand>,
    cmd_tx:   mpsc::Sender<ConductorCommand>,   // cloned out for host to send into
    history:  Vec<Message>,
    plan_id:  Option<PlanId>,
    digest:   SummaryDigestBuffer,
    llm:      LlmClient,
}

pub enum ConductorCommand {
    UserMessage     { text: String, reply_tx: oneshot::Sender<ConductorReply> },
    CriticVerdict   { plan_id: PlanId, version: u32, verdict: Verdict },
    EndConversation,
}

pub enum ConductorReply {
    Chat        { reply: String },
    PlanDrafted { plan_id: PlanId, version: u32 },
    PlanRevised { plan_id: PlanId, version: u32 },
    HotPatched  { task: TaskNode, target_plan: PlanId },
    ScopeChanged{ new_plan_id: PlanId },
    Awaiting    { plan_id: PlanId },
}
```

The host owns the `cmd_tx`, hands it to the TUI/HTTP layer, and drives
the Conductor with `Conductor::run()` which loops on `cmd_rx`.

### 5.2 Loop

```rust
impl Conductor {
    pub async fn run(mut self) -> Result<()> {
        // background: subscribe to bus and feed digest buffer
        let digest = self.digest.spawn_subscriber(self.bus.subscribe());

        while let Some(cmd) = self.cmd_rx.recv().await {
            match cmd {
                ConductorCommand::UserMessage { text, reply_tx } => {
                    let reply = self.handle_user_message(text).await?;
                    let _ = reply_tx.send(reply);
                }
                ConductorCommand::CriticVerdict { plan_id, version, verdict } => {
                    self.handle_critic_verdict(plan_id, version, verdict).await?;
                }
                ConductorCommand::EndConversation => {
                    self.bus.emit(Event::ConversationEnded);
                    break;
                }
            }
        }
        digest.shutdown().await;
        Ok(())
    }
}
```

### 5.3 `handle_user_message` flow

```
1. Pull digest of new Crew summaries since last user message (§5.4),
   inject into LLM context as a system message:
   "While you were away: <digest_markdown>"
2. Classify the message via the cheap classifier prompt (§5.5) if a
   plan is currently running. Otherwise skip the classifier.
3. Branch on classifier output:
     - Informational  → answer in chat, no plan touch
     - HotPatch       → call Captain.inject_task(task), reply ack
     - ScopeChange    → call Captain.pause_all(), draft new plan
                        (overrides current plan_id), emit PlanDrafted
     - NewPlanRequest → draft plan from scratch, emit PlanDrafted
4. If we drafted a plan: validate, save to disk, emit PlanDrafted.
   Critic will fire automatically on PlanDrafted (§6).
```

### 5.4 Digest buffer

The Conductor subscribes to `CrewToolSummary`, `CrewDone`, `CrewFailed`,
`MergeCompleted` events on the bus. A `SummaryDigestBuffer` (in
`digest.rs`) accumulates them in a bounded ring buffer (capacity 512;
older events dropped with a counter).

When the Conductor is next invoked, it calls `digest.drain_markdown()`
which returns something like:

```markdown
While you were away (since last message, 4m12s):

- 47 tool calls across 3 crews
- Crew #2 finished "Add /healthz endpoint" (12 tool calls, merged)
- Crew #1 wrote 14 files, last: src/server/routes.rs
- Crew #3 still running: 23 tool calls in progress
- 2 dropped events (digest buffer overflow)
```

The digest is **not** the raw stream. It is a summary built by
deterministic Rust code (no LLM). Public API:

```rust
pub struct SummaryDigestBuffer { /* ring buffer of last 512 events */ }
impl SummaryDigestBuffer {
    pub fn new(capacity: usize) -> Self;
    pub fn spawn_subscriber(&self, rx: broadcast::Receiver<Event>) -> SubscriberHandle;
    pub fn drain_markdown(&mut self) -> String;
}
```

### 5.5 Mid-run classifier (`prompts/conductor_classifier.md`)

When a user message arrives **and** a plan is currently running, the
Conductor first does a cheap classifier LLM call before responding.
This is a separate, short prompt: it does **not** carry full
conversation history, only:

- The user's new message
- A 1-paragraph summary of the current plan's overview
- A list of in-flight task titles + their statuses

The classifier prompt asks for structured JSON output:

```json
{
  "decision": "informational" | "hot_patch" | "scope_change" | "new_plan",
  "reason":   "<one sentence>",
  "task":     { "title": "...", "description": "...", "deps": ["t3"] }   // present iff hot_patch
}
```

Mapping:

- `informational` — user asked a question. Conductor answers in chat.
  No plan change. Examples: "how's it going?", "what does crew 2 do?"
- `hot_patch` — user wants an additive task that doesn't conflict with
  in-flight work. Conductor calls `Captain.inject_task(task)` (§7.4).
  Captain inserts it into its scheduler at the right dep position.
- `scope_change` — user wants a change that invalidates in-flight
  work. Conductor calls `Captain.pause_all()`, drafts a new plan that
  supersedes the current one, and emits `PlanDrafted` with the new
  plan id (the old plan's `status` becomes `superseded`).
- `new_plan` — no plan was running, or user's ask is unrelated to the
  current one. Conductor drafts from scratch.

**Loop guard:** if a `scope_change` happens, the *new* plan is still
gated by Critic + user approval (§6). The old plan does not resume;
its sandbox is kept for inspection until `ConversationEnded`.

### 5.6 Conductor tool whitelist

| Tool | Notes |
|---|---|
| `read_file`, `list_files`, `grep`, `find` | repo only |
| `skill.*` | any user-defined skill (read-mode by default) |
| `plugin.*` | any installed plugin (read-mode by default; write-mode plugins like `github.create_pr` are denied at planning time) |
| `inject_task_to_captain` | synthetic, internal — calls `Captain.inject_task` |
| `pause_captain` | synthetic, internal |
| `replan` | synthetic, internal — start a new draft plan that supersedes current |

Forbidden at the Conductor tier: shell, write_file, git mutations, http
write methods, docker, k8s, anything that mutates the workspace.

### 5.7 Errors

- `PlanError::Cycle` — conductor produced an invalid DAG. Conductor
  self-corrects (re-prompt with the validation message) up to 3 times,
  then surfaces to user.
- LLM provider error → render one-line summary to chat; conversation
  continues.

System prompt: `mowis-orchestration/src/prompts/conductor.md`.

## 6. `critic.rs` — blind one-shot reviewer

### 6.1 State machine

```
                    ┌─────────────────────────────┐
                    │   Conductor.draft_plan      │
                    └────────────┬────────────────┘
                                 │
                                 ▼
                          Event::PlanDrafted
                                 │
                ┌────────────────┴────────────────┐
                │ in parallel                     │
                ▼                                 ▼
    User UI renders plan              Critic.review_once
    (markdown + diff vs history)      (LLM call, ~5-15s typical)
                │                                 │
                │                                 ▼
                │                       Event::CriticVerdict
                │                                 │
                │                                 ▼
                │                       Conductor receives verdict,
                │                       posts it to chat as a critic
                │                       message:
                │                         "Critic says: <summary>
                │                          Issues: <list>"
                │                                 │
                │                                 ▼
                │                       Conductor decides: revise or stand
                │                       (visible LLM reasoning in chat)
                │                                 │
                │           ┌─────────────────────┤
                │           │                     │
                │      stands as-is          revise → new plan version
                │           │                     │
                │           │                     └──► loop ≤4 versions
                │           ▼
                └────► User sees the latest plan + critic notes
                       inline
                              │
                              ▼
                       User clicks Approve (or types `y`)
                              │
                              ▼
                       Event::UserApproved
                              │
                              ▼
                       Captain spawns / receives plan
```

**Key clarification (v1 got this wrong):** The user is the only
approval gate. The Critic's verdict is **advisory** and flows to the
Conductor. The user sees the Conductor's response to the Critic
inline. The user can click Approve at any time after the plan is
drafted — they do not have to wait for the Critic. (In practice the
Critic completes faster than the user reads the plan, so by the time
the user is reaching for the approve button, the Critic's notes are
already in chat.)

### 6.2 Trigger

The Critic listens for `Event::PlanDrafted`. On each event it spawns
**one** LLM call. Multiple plan versions = multiple Critic invocations.
The Critic instance is stateless; each invocation reads the Plan
directory fresh.

### 6.3 Inputs

- Plan directory's `overview.md`, `tasks.toml`, `sandbox.toml`,
  `models.toml`, `tools.toml`. **Not** the conversation history.
  **Not** the user's raw message. The Critic reviews the Plan
  *as a Plan*, not the user's framing of it.
- A fixed rubric (system prompt) that includes:
  1. Correctness — do the tasks actually accomplish the stated overview?
  2. DAG hygiene — dependencies right? obvious missing tasks?
  3. Safety — does the plan touch production / external side effects
     without explicit user goal?
  4. Tool/model fit — crew tasks budgeted reasonably? flagship models
     used where they shouldn't be?
  5. Sandbox topology — overlays sized appropriately for tasks?

### 6.4 Output: structured verdict

The Critic writes two files to `critic/`:

- `vN.md` — full prose review (what the user sees in chat).
- `vN.toml` — machine-parseable verdict:

```toml
verdict     = "approve"           # approve | revise | block
summary     = "Plan is sound; one minor concern on task ordering."
[[issue]]
severity    = "info"              # info | warn | block
section     = "tasks.toml"
message     = "t3 depends on t2 but t3 doesn't actually need t2's output."
suggested_fix = "Remove t3.deps = [\"t2\"]."

[[issue]]
severity    = "warn"
section     = "models.toml"
message     = "..."
```

The Critic emits `Event::CriticVerdict { plan_id, version, verdict }`.
The Conductor handles all three verdict types in `handle_critic_verdict`:

- `approve` — Conductor posts critic's prose to chat. No revision. User
  may approve whenever they like.
- `revise` — Conductor decides whether to accept the fixes. If yes, it
  revises the plan (new version), snapshots old to `history/vN/`,
  emits `PlanRevised`. Critic re-runs. The user sees both versions and
  the back-and-forth in chat.
- `block` — Conductor posts the block reason prominently. User can
  either accept the block (`UserCancelled`) or explicitly override
  (`UserOverride` which bypasses Critic for this version and routes
  straight to the approval gate).

### 6.5 Loop guard

Hard cap: 4 plan versions per conversation turn. If the Critic still
says `revise` on v4, Conductor stops revising and posts: "Critic and I
disagree; here are both views, you decide."

### 6.6 Public API

## 7. `captain.rs` — execution orchestrator (long-running actor)

### 7.1 Shape

The Captain is an **actor** alive for the whole conversation. It
becomes "active" on first `Event::UserApproved` (it creates the
conversation sandbox at that moment) and stays alive — idle or
executing — until `Event::ConversationEnded`.

```rust
pub struct Captain {
    cfg:           Arc<OrchConfig>,
    bus:           EventBus,
    cmd_rx:        mpsc::Receiver<CaptainCommand>,
    cmd_tx:        mpsc::Sender<CaptainCommand>,
    plan:          Option<Plan>,
    sandbox_id:    Option<String>,
    in_flight:     HashMap<TaskId, CrewHandle>,
    llm:           LlmClient,
    tool_gateway:  ToolGateway,
}

pub enum CaptainCommand {
    // From Conductor / host on Event::UserApproved
    StartPlan        { plan_id: PlanId, reply_tx: oneshot::Sender<()> },

    // From Conductor mid-run (§5.5 hot_patch)
    InjectTask       { task: TaskNode, reply_tx: oneshot::Sender<Result<TaskId>> },

    // From Conductor mid-run (§5.5 scope_change)
    PauseAll         { reply_tx: oneshot::Sender<()> },
    ResumeAll        { reply_tx: oneshot::Sender<()> },
    CancelPlan       { reason: String, reply_tx: oneshot::Sender<()> },

    // From user UI directly (user clicks "Captain panel → query")
    QueryStatus      { reply_tx: oneshot::Sender<CaptainStatus> },

    // From host on Event::ConversationEnded
    Shutdown,
}

pub struct CaptainStatus {
    pub plan_id:      Option<PlanId>,
    pub sandbox_id:   Option<String>,
    pub in_flight:    Vec<(TaskId, String, u32)>,   // (task, agent, tool_calls_so_far)
    pub completed:    Vec<TaskId>,
    pub failed:       Vec<(TaskId, String)>,
}
```

### 7.2 Responsibilities

1. On `StartPlan`: load the (now frozen) Plan via `Plan::load`. If no
   sandbox yet, call `CreateSandbox` on the executor with the plan's
   `sandbox.toml`. Store `sandbox_id` in `status.toml`. Otherwise
   reuse the existing sandbox.
2. Compute task execution order from `tasks.toml` (topological sort,
   schedule in parallel waves where dependencies allow).
3. For each ready task: `CreateAgentOverlay { sandbox_id, agent_id }`,
   spawn a `Crew` (§8) on a tokio task with the task spec, agent_id,
   and the model resolved from `models.toml`. Crews emit
   `CrewToolSummary` per tool call (§8.3).
4. As Crew tasks complete:
   - `Done` → merge their overlay (§7.7).
   - `Failed` → consult retry policy (§7.6).
5. On `InjectTask` (mid-run): validate the new task's deps reference
   either completed or in-flight tasks; insert into the scheduler;
   spawn Crew when ready.
6. On `PauseAll`: send pause signal to every in-flight Crew (Crew
   completes its current LLM round, then halts). On `ResumeAll`:
   re-send tasks. On `CancelPlan`: cancel all in-flight, discard
   overlays, write final status, emit `PlanFailed { reason }`.
7. On `Shutdown`: cancel any in-flight crews, call `DestroySandbox`,
   exit.

The Captain is **also an LLM loop** — it has a Sonnet-tier model and
the tools listed in §7.3. The LLM is consulted when the Captain needs
judgement: should we retry this Crew? was the merge clean enough?
should we ask the Conductor to replan a subgraph? The deterministic
scheduling above is the default; the LLM is the fallback / escalation
path.

### 7.3 Tools the Captain can call

Synthetic, dispatched to methods on `Captain` (not vsock) — except
overlay/sandbox ones which go to the executor:

```
- start_crew(task_id)
- get_crew_status(agent_id)
- pause_crew(agent_id)
- cancel_crew(agent_id)
- merge_overlay(agent_id)
- discard_overlay(agent_id)
- replan_subgraph(root_task_id, reason)
```

No filesystem read/write, no shell, no git, no plugins. The Captain
LLM cannot touch user files.

### 7.4 `inject_task` semantics


### 7.6 Retry policy

Per task:
- Tool failure inside a Crew → Crew retries locally up to 3 times
  (already in `agent_execution.rs`).
- Crew exits with `Failed` → Captain retries up to **2 times** with the
  same agent overlay (discarding previous). On 3rd failure, Captain LLM
  decides: `replan_subgraph` (asks Conductor to redraft this task and
  its downstream) or `abort`.

### 7.7 Merge policy

- Sibling tasks (no deps between them) merge in dependency-tree order
  using `merge_worker.rs`'s tree-merge (port that module — see §14.2).
- Two-way merge first (overlay vs current sandbox state). On conflict,
  Captain LLM is consulted with both file versions and the task
  descriptions, and produces a 3-way resolution. Cap conflict-repair
  attempts at 3.
- After all merges, the sandbox represents the final desired state.

### 7.8 Conversation end

On `Event::ConversationEnded` (emitted by Conductor on `EndConversation`
command, which the host sends on user "/end" or session close):

1. Cancel all in-flight crews.
2. Call `MergeAgentOverlay` for any clean-exit overlays, `DiscardAgentOverlay`
   for the rest.
3. Call `DestroySandbox(sandbox_id)` on the executor.
4. Emit `Event::CaptainShutdown { sandbox_id, final_plan_status }`.
5. Captain's `run()` loop exits.

The conversation sandbox is **not** retained after `ConversationEnded`.
If the user wants to inspect, they ask before sending `EndConversation`
(Captain.QueryStatus is available throughout).

### 8.2 Loop

```rust
loop {
    round = call_agent_round(model, conversation, tool_schemas).await?;
    if round.is_final() { break }
    for tool_call in round.tool_calls {
        if !whitelist.allows(&tool_call.name) {
            let outcome = ToolOutcome::Denied;
            let summary = summarize(&tool_call, &outcome);
            bus.emit(Event::CrewToolSummary {
                agent_id: agent_id.clone(),
                text: summary, tool_name: tool_call.name.clone(),
                success: false,
            });
            conversation.push_tool_results(vec![(tool_call, json!({"error": "tool forbidden"}))]);
            continue;
        }
        let outcome = tool_gateway.invoke(sandbox_id, agent_id, &tool_call).await;
        checkpoint::save(&conversation);

        let summary = summarize(&tool_call, &outcome);
        bus.emit(Event::CrewToolSummary {
            agent_id: agent_id.clone(),
            text: summary, tool_name: tool_call.name.clone(),
            success: outcome.is_ok(),
        });

        conversation.push_tool_results(vec![(tool_call, outcome.into_value())]);
    }
    if conversation.rounds >= task.tool_budget { break with budget_exceeded }
}
emit Event::CrewDone { agent_id, summary } // or CrewFailed
```

### 8.3 The summary contract

For every tool call (allowed or denied, success or failure), exactly
one `CrewToolSummary` event is emitted. Text is **deterministic** —
generated by `summaries::summarize` from the `ToolCall` and
`ToolOutcome`, no LLM cost. See §12.4 for templates.

Example texts:

- `"Agent ran a terminal command: npm install"`
- `"Agent ran `npm install` (FAILED: exit code 1)"`
- `"Agent read src/server/routes.rs (4.2 KB)"`
- `"Agent wrote 47 lines to src/server/routes.rs"`
- `"Agent grepped for 'fn main' in src/ (3 matches in 2 files)"`
- `"Agent committed: 'feat: add /healthz endpoint' (3 files)"`
- `"Agent fetched https://api.example.com/v1/health (200 OK, 1.1 KB)"`
- `"Agent ran docker container: alpine:3.19 sh -c '...'"`

## 12. `tools.rs` + `summaries.rs` — tier whitelists, ToolGateway, summary templates

### 12.1 Conductor tool whitelist

| Tool | Notes |
|---|---|
| `read_file` | local repo only |
| `list_files` | local repo only |
| `grep` / `find` | local repo only |
| `skill.*` | any user-defined skill (read-mode) |
| `plugin.*` | any installed plugin (read-mode only) |

Forbidden: shell, write_file, git mutations, http write methods, docker,
k8s.

The Conductor reads the repo from the host filesystem directly (no
vsock) — its read-only tools short-circuit in `ToolGateway` and read
from `{repo_root}` on the host. This is the only host-local tool
execution; everything else crosses vsock.

### 12.2 ToolGateway

```rust
pub struct ToolGateway {
    transport: Arc<Mutex<VsockTransport>>,   // re-uses mowis-host/src/transport
    tier:      Tier,
    allowlist: ToolAllowlist,
    repo_root: PathBuf,                      // for Conductor short-circuit
}

impl ToolGateway {
    pub async fn invoke(&self,
                        sandbox_id: &str,
                        agent_id:   &str,         // for crew; empty for conductor
                        call:       &ToolCall) -> Result<ToolOutcome>;
}

pub enum ToolOutcome {
    Ok(serde_json::Value),
    Err(String),
    Denied,
}
```

The gateway:
1. Checks `self.allowlist.allows(&call.name)` → if not, returns
   `ToolOutcome::Denied` (LLM sees as tool error, not panic).
2. If tier is Conductor and the tool is in the short-circuit set
   (read_file, list_files, grep, find), execute locally against
   `repo_root`. No vsock.
3. Otherwise translate the `ToolCall` into `InvokeToolAsAgent` and
   round-trip over vsock.
4. Await response with a per-tier timeout (Crew: 5 min default;
   Conductor: 30s).

### 12.3 Critic — no gateway

The Critic has no `ToolGateway`. Its constructor doesn't even take one.
Build-time guarantee.

### 12.4 Summary registry (`summaries.rs`)

Deterministic, no LLM. One function: `summarize(call, outcome) -> String`.

```rust
pub fn summarize(call: &ToolCall, outcome: &ToolOutcome) -> String {
    let success = matches!(outcome, ToolOutcome::Ok(_));
    let err_tail = match outcome {
        ToolOutcome::Err(e)  => format!(" (FAILED: {})", truncate(e, 60)),
        ToolOutcome::Denied  => " (DENIED by tool whitelist)".to_string(),
        ToolOutcome::Ok(_)   => "".to_string(),
    };

    match call.name.as_str() {
        "shell_exec" => {
            let cmd = call.args.get("cmd").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            format!("Agent ran a terminal command: `{}`{}", truncate(cmd, 80), err_tail)
        }
        "read_file" => {
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            let size = outcome.ok_value()
                .and_then(|v| v.get("size_bytes"))
                .and_then(|v| v.as_u64())
                .map(human_bytes)
                .unwrap_or_else(|| "?".into());
            if success { format!("Agent read {} ({})", path, size) }
            else { format!("Agent tried to read {}{}", path, err_tail) }
        }
        "write_file" => {
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            let lines = call.args.get("content").and_then(|v| v.as_str())
                .map(|s| s.lines().count()).unwrap_or(0);
            if success { format!("Agent wrote {} lines to {}", lines, path) }
            else { format!("Agent tried to write to {}{}", path, err_tail) }
        }
        "list_files" => {
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or(".");
            let count = outcome.ok_value()
                .and_then(|v| v.get("entries"))
                .and_then(|v| v.as_array())
                .map(|a| a.len())
                .unwrap_or(0);
            if success { format!("Agent listed {} entries in {}", count, path) }
            else { format!("Agent tried to list {}{}", path, err_tail) }
        }
        "grep" | "find" => {
            let pattern = call.args.get("pattern").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or(".");
            let matches = outcome.ok_value()
                .and_then(|v| v.get("match_count"))
                .and_then(|v| v.as_u64())
                .unwrap_or(0);
            if success {
                format!("Agent searched for `{}` in {} ({} matches)",
                        truncate(pattern, 40), path, matches)
            } else {
                format!("Agent tried to search for `{}`{}", pattern, err_tail)
            }
        }
        "git_commit" => {
            let msg = call.args.get("message").and_then(|v| v.as_str()).unwrap_or("");
            let files = outcome.ok_value()
                .and_then(|v| v.get("files_changed"))
                .and_then(|v| v.as_u64())
                .unwrap_or(0);
            if success { format!("Agent committed: '{}' ({} files)", truncate(msg, 50), files) }
            else { format!("Agent tried to commit{}", err_tail) }
        }
        "git_diff" => {
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or("(repo)");
            format!("Agent diffed {}{}", path, err_tail)
        }
        "http_get" | "http_post" | "http_put" | "http_delete" => {
            let url = call.args.get("url").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            let method = call.name.strip_prefix("http_").unwrap().to_uppercase();
            let status = outcome.ok_value()
                .and_then(|v| v.get("status"))
                .and_then(|v| v.as_u64())
                .map(|s| format!(" ({})", s))
                .unwrap_or_default();
            if success { format!("Agent fetched {} {}{}", method, truncate(url, 60), status) }
            else { format!("Agent tried to fetch {} {}{}", method, url, err_tail) }
        }
        "docker_run" => {
            let image = call.args.get("image").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            let cmd = call.args.get("cmd").and_then(|v| v.as_str()).unwrap_or("");
            format!("Agent ran docker container: {} {}{}", image, truncate(cmd, 40), err_tail)
        }
        "apply_patch" => {
            let path = call.args.get("path").and_then(|v| v.as_str()).unwrap_or("<unknown>");
            if success { format!("Agent applied patch to {}", path) }
            else { format!("Agent failed to apply patch to {}{}", path, err_tail) }
        }
        "k8s_apply" => {
            let manifest = call.args.get("manifest").and_then(|v| v.as_str()).unwrap_or("<inline>");
            format!("Agent applied k8s manifest: {}{}", manifest, err_tail)
        }
        // Fallback for any tool without an explicit template:
        other => format!("Agent invoked `{}`{}", other, err_tail),
    }
}

fn truncate(s: &str, n: usize) -> String { /* … */ }
fn human_bytes(n: u64) -> String { /* "4.2 KB", "1.1 MB" */ }
```

The default fallback ensures every tool — even ones we haven't added a
template for yet — produces a sane summary. As we add tools we add
templates; the fallback is the safety net.

### 12.5 Captain tool whitelist

Captain's tools are **internal**, not LLM-arbitrary. The Captain's LLM
sees a small synthetic schema (§7.3). Each dispatches to a method on
`Captain`, not over vsock (except `merge_overlay`/`discard_overlay`
which do).

### 12.6 Crew tool whitelist

Default-allow everything. Apply deltas from `tools.toml` (§4.6).
Server-side enforcement: every `InvokeToolAsAgent` carries the caller's
tier in the `caller_tier` field (§11). The executor rejects forbidden
combinations even if the host's gateway is buggy.

## 13. UI streams — what the user sees, what they have to click for

### 13.1 Three visibility tiers

| Stream | Default visibility | Source | Contents |
|---|---|---|---|
| **Conductor chat** | **Always visible** (main pane) | `Event::ConductorReply`, `Event::CriticVerdict`, user messages, plan drafts/links | The whole conversation — what you'd see in Claude Code's chat pane. Critic verdicts appear as a labelled critic message; plan drafts appear as a link or inline preview. |
| **Captain panel** | Collapsed by default (click "Captain" tab to expand) | `Event::CrewToolSummary`, `Event::CrewStarted`, `Event::CrewDone`, `Event::CrewFailed`, `Event::MergeCompleted`, `Event::TaskInjected`, `Event::CaptainStatusUpdate` | One line per event. Human-readable. The "what is being done right now" stream. |
| **Per-agent drill-down** | Collapsed inside Captain panel (click an agent row to expand) | Same as Captain panel, filtered to a single `agent_id` + optional raw tool I/O on demand | Per-crew detailed trail. Clicking a row reveals the raw tool input/output (lazy-fetched from `status.toml` + on-disk tool logs). |

### 13.2 Where the streams come from

The host's TUI/HTTP layer subscribes to the event bus. Each UI stream
is just a filter:

```rust
// Pseudocode for the UI subscriber
let mut rx = bus.subscribe();
while let Ok(ev) = rx.recv().await {
    match ev {
        Event::ConductorReply { .. } | Event::CriticVerdict { .. } |
        Event::UserMessageReceived { .. } | Event::PlanDrafted { .. } |
        Event::PlanRevised { .. } => ui.push_chat(ev),

        Event::CrewToolSummary { .. } | Event::CrewStarted { .. } |
        Event::CrewDone { .. } | Event::CrewFailed { .. } |
        Event::MergeCompleted { .. } | Event::TaskInjected { .. } |
        Event::CaptainStatusUpdate { .. } => ui.push_captain(ev),

        _ => ui.push_audit(ev),   // hidden audit log
    }
}
```

### 13.3 Raw tool I/O on demand

The summary stream is what flows on the bus. Raw tool input/output
(the actual stdout/stderr blob, the full diff, the HTTP response body)
is **persisted per call** to
`.mowis/plans/<plan_id>/tool_logs/<agent_id>/<round>.json` by the
Crew. The UI fetches these lazily when the user expands a row. They
are never broadcast on the bus.

### 13.4 v1 (stdin REPL) scope

For the initial CLI cutover, `mowisd chat` is a stdin REPL. The three
tiers are rendered as:

- **Chat stream:** printed to stdout as it arrives.
- **Captain stream:** printed prefixed with `[captain]` only if the
  user types `/captain on` (off by default; matches the "click to
  reveal" model).
- **Per-agent drill-down:** `mowisd chat tail-agent <agent_id>` in
  a second terminal.

Full TUI integration happens after the cutover.

## 17. Hard rules (do not violate)

1. **No tool runs in the host process** except the Conductor read-only
   short-circuit (§12.2 item 2). Every other tool dispatches over
   vsock. The Captain's synthetic orchestration tools (§7.3) are
   method calls on Captain, not tool calls.
2. **Critic is stateless and blind.** Never sees conversation history,
   only the Plan dir. Constructor cannot accept a `ToolGateway` or any
   reference to user messages.
3. **The Plan is the source of truth.** The bus is informational; if
   the host crashes mid-execution, recovery reads `.mowis/plans/<id>/`
   and `status.toml`.
4. **No `unwrap()` in production code paths.** Use `?` or proper error
   handling (CLAUDE.md invariant).
5. **No direct crew-to-crew communication.** Crews only see their own
   overlay. Cross-task data flow happens by Captain merging overlays.
6. **`mowis-protocol` wire version bumps to 2** in §16.2. Both host
   and executor refuse to talk to a peer with a different
   `PROTOCOL_VERSION`.
7. **No bypass of the tool whitelist.** Both bind-time (ToolGateway)
   and server-side (executor `caller_tier` check) enforcement must
   agree. If they disagree, that's a bug.
8. **All TOML in plan dirs is atomic** (write tmp + rename).
9. **Every Crew tool call emits exactly one `CrewToolSummary`** —
   success, failure, or denied. No exceptions, no batching.
10. **Conductor never reads the raw summary stream** — only the
    digest. Capacity-bounded ring buffer in `digest.rs` is the only
    way `CrewToolSummary` enters the Conductor LLM context.
11. **Tests must pass before each section is considered done.**
12. **Never delete or modify tests to make them pass** (CLAUDE.md).
